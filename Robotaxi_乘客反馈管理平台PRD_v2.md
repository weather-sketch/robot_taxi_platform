# Robotaxi 乘客反馈管理平台 PRD v2

---

# 一、项目背景

随着 Robotaxi 规模化运营，用户反馈快速增长，当前主要问题：

- 投诉分散，缺乏统一管理入口
- 处理流程不规范，进度不可追踪
- 难以识别高优问题与系统性问题
- 数据无法有效支撑产品优化

需要搭建一个平台，实现：

**反馈管理 → 工单处理 → 数据分析 → AI 洞察 的闭环系统**

---

# 二、产品目标

## 2.1 核心目标

1. 提供标准化反馈管理能力
2. 建立工单处理体系，闭环问题处理
3. 提供数据统计能力，支撑运营决策
4. 引入 AI 能力，降低分析成本并提升洞察能力
5. 轻量对接外部舆情信号，辅助风险感知

## 2.2 量化指标

| 指标 | 当前基线 | V1 目标 |
|------|---------|--------|
| 平均投诉处理时长 | 72h | ≤ 24h |
| SLA 达标率 | 60% | ≥ 90% |
| P0/P1 工单响应时间 | 无追踪 | 100% 在 SLA 时限内响应 |
| AI 自动分类准确率 | — | ≥ 85% |

> 注：基线值为估算值，上线后第一个月以实际数据校准。

---

# 三、产品结构

系统分为四大模块：

1. **反馈列表与管理**（含工单体系）
2. **数据统计仪表盘**
3. **AI 智能分析**（分类 / 聚类 / 摘要 / 建议 / 优先级评估）
4. **轻量舆情监控**（外部信号对接）

# 五、模块设计

---

## 5.1 模块一：反馈列表与管理

### 5.1.1 功能定位

从"反馈查看工具"升级为：

**反馈驱动的轻量工单系统**

支持：查看 → 分类 → 建单 → 分派 → 跟进 → 关闭

---

### 5.1.2 核心功能

#### （1）反馈列表

支持：

- 分页展示
- 排序：时间（默认倒序） / 评分
- 筛选：
    - 评分范围
    - 时间范围
    - 路线
    - 反馈类型
    - 反馈来源
    - 工单状态
    - 优先级

#### （2）反馈详情页

展示：

- **基础信息**：用户ID / 行程ID / 车辆ID / 评分 / 反馈文本 / 反馈时间
- **行程信息**：路线 / 城市 / 时间 / 时长
- **反馈分类**：AI 自动分类结果 + 人工修正（类型见 5.3.2）
- **工单信息**：状态 / 优先级 / 负责人 / 处理记录 / SLA 状态
- **关联信息**：相似反馈列表（由 AI 模块提供，见 5.3.3）

#### （3）反馈来源

| 来源标识 | 说明 |
|---------|------|
| `app_rating` | App 内行程评价 |
| `app_feedback` | App 内意见反馈 |
| `customer_service` | 客服渠道 |
| `social_media` | 外部舆情抓取（见模块四） |

---

### 5.1.3 工单体系

#### （1）工单生成

支持两种方式：

- **手动创建**：运营判断需要处理时手动创建
- **自动创建**：规则触发（如低评分 ≤ 2 分、AI 标记 P0/P1）

#### （2）工单字段

| 字段 | 说明 |
|------|------|
| 工单 ID | 系统自动生成 |
| 关联反馈 ID | 关联触发该工单的反馈（1:1 关系） |
| 优先级 | P0 / P1 / P2 / P3 |
| 状态 | 待处理 / 处理中 / 已解决 / 已关闭 |
| 负责人 | 指定处理人 |
| 创建时间 | 自动记录 |
| 更新时间 | 自动记录 |
| SLA 截止时间 | 根据优先级自动计算 |
| SLA 状态 | 正常 / 即将超时 / 已超时 |
| 处理方式 | 补偿 / 技术问题 / 无需处理 / 其他 |
| 处理备注 | 自由文本 |

#### （3）优先级定义与 SLA

| 优先级 | 定义 | 响应时限 | 解决时限 |
|--------|------|---------|---------|
| P0 | 安全/SOS 类问题 | 1 小时 | 4 小时 |
| P1 | 严重体验问题 | 1 小时 | 24 小时 |
| P2 | 一般差评 | 4 小时 | 72 小时 |
| P3 | 建议类反馈 | 24 小时 | 7 天 |

优先级来源：
- **AI 自动标记**：规则引擎 + LLM 辅助判定（见 5.3.3）
- **人工调整**：运营可手动覆盖 AI 结果

#### （4）工单流转

```
创建工单 → 分派负责人 → 进入处理中 → 更新处理记录 → 完成并关闭
                ↑                                    │
                └──── 需转派时退回重新分派 ←───────────┘
```

每次状态变更自动记录操作日志（操作人、时间、变更内容）。

#### （5）通知与升级机制

| 事件 | 通知对象 | 通知渠道 |
|------|---------|---------|
| P0 工单创建 | 负责人 + 运营主管 | 企业微信 + 站内信 |
| P1 工单创建 | 负责人 | 企业微信 + 站内信 |
| 工单分派/转派 | 新负责人 | 站内信 |
| SLA 即将超时（剩余 25%） | 负责人 | 企业微信提醒 |
| SLA 已超时 | 负责人 + 运营主管 | 企业微信 + 升级标记 |
| SLA 超时 >2 倍时限 | 运营主管上级 | 自动升级通知 |

### 5.1.4 增强功能

#### （1）用户维度聚合

- 查看同一用户的所有反馈历史
- 辅助判断重复问题与高频投诉用户

#### （2）批量操作

- 同类问题批量建单 / 批量分派

#### （3）车辆维度聚合

- 查看同一车辆关联的所有反馈
- 辅助定位问题车辆

## 5.2 模块二：数据统计仪表盘

### 5.2.1 功能定位

提供整体反馈与工单处理情况的**数据视角**。

### 5.2.2 核心功能

#### （1）核心指标卡

- 总反馈数（今日 / 本周 / 本月）
- 平均评分
- 好评率（≥ 4 分占比） / 差评率（≤ 2 分占比）
- 未关闭工单数
- SLA 达标率

#### （2）趋势图

- 差评数量趋势
- 好评率趋势
- 平均评分趋势

支持按日 / 周 / 月切换粒度。

#### （3）分布分析

- 按评分分布
- 按路线分布
- 按时间段分布（高峰 vs 平峰）
- 按反馈类型分布
- 按城市分布

#### （4）工单处理指标

- 各优先级工单数量
- 平均处理时长（按优先级拆分）
- SLA 达标率（按优先级拆分）
- 未关闭工单数及老化分布

### 5.2.3 进阶能力

#### （1）Top 问题排行

基于 AI 聚类结果（见 5.3.3），展示当前最高频问题 Top N。

#### （2）异常波动检测

检测逻辑：
- **规则型**：某路线/时段/城市差评率环比上升 > 20% 时触发告警
- **基线型**：评分偏离近 30 天均值 ± 2 个标准差时触发告警

告警方式：仪表盘醒目标记 + 企业微信通知运营主管。

#### （3）A/B 实验维度筛选

- 支持按实验分组 ID 筛选反馈数据
- 对比不同分组的评分、好评率、问题分布
- 前提：反馈数据需携带实验分组标识（由上游系统写入）

> 注：本平台不负责实验管理（分流、分组），仅作为实验数据的消费方。

## 5.3 模块三：AI 智能分析

### 5.3.1 功能定位

利用大模型实现：

**自动理解反馈 → 分类标记 → 聚类问题 → 输出洞察与建议**

### 5.3.2 核心功能

#### （1）自动分类

反馈入库后异步触发 AI 分类，输出以下类别：

| 分类 | 说明 |
|------|------|
| 驾驶行为 | 急刹、急转、速度不当、变道犹豫、起步慢等 |
| 乘坐舒适度 | 减震差、空调、座椅、噪音、异味、音响等 |
| 车内环境 | 卫生、设备损坏、污渍等 |
| 路线规划 | 绕路、路线不合理等 |
| 安全感知 | 感觉不安全、险情、SOS 等 |
| 费用相关 | 计费异常、价格不满等 |
| 新用户引导 | 找不到车、不会操作等 |
| 其他 | 无法归入以上类别 |

分类结果展示在反馈详情页，支持人工修正。

#### （2）摘要生成

对多条反馈（如同一聚类下）生成：

- 核心问题总结（1-2 句话）
- 高频用户表达（原文代表句）
- 涉及反馈条数与时间范围

#### （3）产品建议生成

基于聚类分析结果输出：

- 问题总结
- 影响范围评估（涉及路线、城市、时间段）
- 优化建议

### 5.3.3 AI 增强能力

#### （4）相似反馈聚类

将反馈 Embedding 后通过向量相似度自动聚合为"问题集群"：

- 识别高频重复问题
- 发现系统性问题（同一路线/时段/车辆的集中投诉）

输出：
- Top 问题集群列表
- 每个集群对应的反馈原文集合
- 集群标签（AI 自动生成）

在反馈详情页展示"相似反馈"列表，辅助运营判断。

#### （5）优先级自动评估

评估逻辑（分层）：

1. **规则引擎（确定性）**：
    - 包含安全/SOS 关键词 → P0
    - 评分 ≤ 1 分 → 至少 P1
    - 评分 = 2 分 → 至少 P2
2. **LLM 辅助（模糊场景）**：
    - 对规则未覆盖的反馈，LLM 分析严重程度
    - 输出建议优先级，人工可覆盖

#### （6）AI 分析报告

定期（每周/每月）自动生成分析报告：
- 本期 Top 问题汇总
- 与上期对比变化
- 优化建议

## 5.4 模块四：轻量舆情监控

### 5.4.1 功能定位

对接外部社交媒体信号，作为内部反馈数据的**补充视角**，辅助风险感知。

> 注：本模块定位为轻量对接，不自建爬虫体系。优先对接公司已有的舆情监控平台或第三方舆情 SaaS 服务的 API。

### 5.4.2 核心功能

#### （1）外部信号接入

数据来源：小红书 / 微博 / 抖音（通过舆情平台 API 获取已结构化的数据）

接入方式：
- **优先**：对接公司内部已有舆情系统，消费其输出结果
- **备选**：对接第三方舆情 SaaS（如清博、新榜等）API

接入数据字段：
| 字段 | 说明 |
|------|------|
| 平台 | 微博 / 小红书 / 抖音 |
| 内容摘要 | 原文或平台提供的摘要 |
| 情感倾向 | 正面 / 负面 / 中性（由舆情平台输出） |
| 热度指标 | 转发/点赞/评论数 |
| 抓取时间 | 数据入库时间 |
| 原文链接 | 跳转至原始内容 |

#### （2）舆情信号展示

- 独立的舆情信号列表页（与内部反馈列表分离）
- 支持按平台、情感倾向、时间范围筛选
- 负面内容醒目标记

#### （3）负面趋势告警

- 负面舆情数量突增时触发告警（阈值可配置）
- 通知运营主管关注

#### （4）人工关联

- 运营可手动将某条舆情信号关联到内部反馈或工单
- 辅助判断外部舆情与内部问题的关联性

### 5.4.3 边界说明

本模块**不负责**：
- 自建爬虫或数据采集
- 社媒内容的 NLP 分析（依赖上游舆情平台）
- 自动回复或社媒互动

# 六、核心流程

## 6.1 标准反馈处理流程

```
用户反馈进入系统
    │
    ▼
AI 自动分类 + 优先级评估
    │
    ▼
是否需要建单？──── 否 ───▶ 归档，纳入统计
    │
    是
    ▼
创建工单（自动或手动）
    │
    ▼
分派负责人 ──▶ 通知负责人
    │
    ▼
跟进处理 ──▶ 更新处理记录
    │
    ▼
完成并关闭 ──▶ 纳入统计
```

## 6.2 高优问题快速响应流程

```
检测到安全/高风险关键词 或 AI 标记 P0/P1
    │
    ▼
自动创建工单 + 自动标记高优
    │
    ▼
即时通知负责人 + 运营主管（企业微信）
    │
    ▼
SLA 倒计时启动
    │
    ▼
SLA 即将超时 ──▶ 提醒通知
    │
    ▼
SLA 已超时 ──▶ 自动升级至上级
```

## 6.3 系统性问题发现流程

```
AI 聚类发现某问题集群反馈量激增
    │
    ▼
仪表盘 Top 问题排行更新
    │
    ▼
异常波动检测触发告警
    │
    ▼
运营/产品查看 AI 摘要与建议
    │
    ▼
创建关联工单，推动产品/技术修复
```

# 七、产品边界

核心职责：

**反馈管理 + 工单闭环 + 数据分析 + AI 洞察 + 轻量舆情**

| 负责 | 不负责 |
|------|--------|
| 内部反馈的全生命周期管理 | 用户端反馈入口的UI设计 |
| 工单创建、流转、关闭 | 客服系统（电话/在线客服） |
| AI 分类、聚类、摘要、建议 | 模型训练与调优 |
| 数据统计与可视化 | BI 平台能力 |
| 消费外部舆情平台数据 | 自建爬虫或社媒数据采集 |
| 按实验分组筛选反馈 | A/B 实验管理（分流/分组） |
